Optimizing Websites for SEO and Speed

Introduction

A fast, well-structured website is vital. Users expect pages that load quickly and search engines reward sites that are accessible, fast, and properly marked up. In this lesson you’ll learn practical, hands-on techniques for improving both SEO (Search Engine Optimization) and performance.

We’ll cover metadata and structured data, content & headings, images and media, font loading, HTTP optimizations, caching, CDNs, build steps (minify/concat), measuring performance, and launch checklists.

How to use this lesson: Copy examples into your project, test with your dev tools and Lighthouse, then iterate. Each code block has a Copy button.

1 — SEO Basics: Titles, Descriptions, and Canonicals

Every page needs a unique <title> and a descriptive meta description. These help search engines and appear in search results. Add a canonical URL to avoid duplicate content issues. 

<!-- Head of each page -->
<title>Services — ACME Web Design</title>
<meta name="description" content="Professional website design and development for small businesses. Fast, responsive & optimized." />
<link rel="canonical" href="https://example.com/services.html" />

Tips: Keep titles ~50–60 characters and meta descriptions ~140–160 characters. Make them useful and unique per page.

2 — Structured Data (JSON-LD)

Use structured data (Schema.org) to give search engines explicit information about your pages — business info, articles, breadcrumbs, events, etc. Google prefers JSON-LD placed in the head. 

<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Organization",
  "name": "ACME Web Design",
  "url": "https://example.com",
  "logo": "https://example.com/images/logo.png",
  "sameAs": [
    "https://www.facebook.com/acme",
    "https://twitter.com/acme"
  ]
}
</script>

Common structured data: Organization, LocalBusiness, Article, BreadcrumbList, Product. Test with Google’s Rich Results Test or structured data testing tools.

3 — Sitemap & robots.txt

Provide a sitemap.xml to help crawlers discover your pages. Use robots.txt to control crawler access to certain resources. 

<!-- sitemap.xml (place at site root) -->
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url>
    <loc>https://example.com/</loc>
    <lastmod>2025-10-15</lastmod>
    <priority>1.00</priority>
  </url>
  <url>
    <loc>https://example.com/services.html</loc>
    <lastmod>2025-09-10</lastmod>
    <priority>0.80</priority>
  </url>
</urlset>

# robots.txt (site root)
User-agent: *
Disallow: /private/
Sitemap: https://example.com/sitemap.xml

Tip: Submit your sitemap in Google Search Console and Bing Webmaster Tools after deployment.

4 — Content Structure & Headings

Organize content with semantic HTML: <h1> for the page title, then <h2>, <h3> for sections. Good headings improve accessibility and SEO. 

<main>
  <h1>Professional Web Design</h1>
  <section>
    <h2>Our Services</h2>
    <p>We build responsive websites for small businesses.</p>
  </section>
</main>

Do not skip heading levels (e.g., don’t jump from <h1> to <h4>), and ensure your main content is crawlable in HTML (not only via JS-rendered content).

5 — Images & Media Optimization

Images are often the largest assets. Optimize them: serve responsive sizes, modern formats (WebP), compress, lazy-load, and provide width/height attributes to avoid layout shift.

Responsive images with srcset

<img
  src="/images/hero-800.jpg"
  srcset="/images/hero-400.jpg 400w, /images/hero-800.jpg 800w, /images/hero-1200.jpg 1200w"
  sizes="(max-width:600px) 100vw, 800px"
  alt="Team designing website"
  loading="lazy"
  width="1200" height="600">

Use modern formats

<picture>
  <source type="image/webp" srcset="/images/hero-800.webp 800w, /images/hero-1200.webp 1200w">
  <img src="/images/hero-800.jpg" alt="Team designing website" loading="lazy" width="1200" height="600">
</picture>

Tools: Use image compressors (Squoosh, ImageMagick) or build tools to generate multiple sizes automatically. Always include descriptive alt text for accessibility and SEO.

6 — Font Loading & Optimization

Web fonts improve design but can block rendering. Use best practices: preconnect/preload, display swap, and limit font variants to reduce size. 

<!-- Preconnect to Google Fonts -->
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link href="https://fonts.googleapis.com/css2?family=Inter:wght@400;600&display=swap" rel="stylesheet">

/* CSS */
body { font-family: 'Inter', system-ui, Arial, sans-serif; font-display: swap; }

Notes: Use display=swap to avoid invisible text (FOIT). When possible self-host fonts and subset them to only required characters to improve performance.

7 — Critical CSS & Render-Blocking Resources

CSS is render-blocking by default. Extract critical CSS for above-the-fold content and defer the rest. Tools like Penthouse or build plugins can help automate critical CSS extraction. 

<!-- Inline small critical CSS in the head -->
<style>
/* minimal critical styles for header & hero */
body{margin:0;font-family:Inter,system-ui}
header{background:#007bff;color:#fff;padding:18px}
</style>
<link rel="stylesheet" href="/css/main.css" media="print" onload="this.media='all'">
<noscript><link rel="stylesheet" href="/css/main.css"></noscript>

This technique ensures content renders quickly while the main stylesheet loads non-blocking.

8 — JavaScript Loading Strategies

Large JS files can slow initial load. Defer non-critical scripts and load essential code asynchronously. Split code into smaller chunks for better caching. 

<!-- Place at end of body or use defer/async -->
<script src="/js/vendor.js" defer></script>
<script src="/js/main.js" defer></script>

When to use: Use defer for scripts that rely on DOM and should execute in order. Use async for independent third-party scripts (analytics) that can run anytime.

9 — Caching & CDN

Use caching headers and a CDN to distribute static assets globally. Proper cache-control rules reduce repeat-load times significantly.

Nginx example — caching headers

location ~* \.(js|css|png|jpg|jpeg|webp|svg|gif)$ {
  expires 30d;
  add_header Cache-Control "public, max-age=2592000, immutable";
}

CDN: Use Cloudflare, Fastly, or your hosting CDN to serve assets from edge locations. Many static hosts (Netlify, Vercel) include CDN distribution automatically.

10 — Compression & HTTP/2

Enable gzip or Brotli compression on the server to reduce asset sizes. Use HTTP/2 for multiplexing requests over a single connection to improve parallel loading. 

# Nginx example: enable gzip
gzip on;
gzip_types text/plain text/css application/json application/javascript text/xml application/xml application/xml+rss text/javascript;
gzip_min_length 256;

Note: Brotli often compresses better than gzip; enable it where supported. Also prefer HTTPS — HTTP/2 requires TLS on most browsers.

11 — Preconnect, Prefetch & Preload

Hint the browser about important resources using preconnect, preload, and prefetch. 

<!-- Preconnect to font host for faster font load -->
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

<!-- Preload hero image for fast LCP -->
<link rel="preload" as="image" href="/images/hero-1200.webp">

Use carefully: Overusing preload/preconnect can waste bandwidth. Only hint the most critical resources (fonts, hero images, critical scripts).

12 — Progressive Enhancements & Service Worker

Consider adding a service worker for caching and offline support (PWA). Start simple — cache static assets and return them for subsequent visits. 

// simple sw.js (service worker)
self.addEventListener('install', event => {
  event.waitUntil(
    caches.open('site-v1').then(cache => cache.addAll([
      '/',
      '/index.html',
      '/css/main.css',
      '/js/main.js',
      '/images/hero-800.webp'
    ]))
  );
});

self.addEventListener('fetch', event => {
  event.respondWith(
    caches.match(event.request).then(resp => resp || fetch(event.request))
  );
});

Remember: Service workers require HTTPS and careful cache versioning during updates to avoid stale content issues.

13 — Build Steps: Minify, Bundle & Tree-Shake

Use build tools to minify CSS/JS, bundle modules, and remove unused code. Tools: Vite, esbuild, Rollup, Webpack. Minification reduces file sizes; tree-shaking removes dead code. 

{
  "scripts": {
    "build": "vite build",
    "dev": "vite"
  },
  "devDependencies": {
    "vite": "^4.0.0"
  }
}

Also: Compress build outputs (gzip/brotli) before uploading to some CDNs or enable server compression on the host.

14 — Measure: Lighthouse, WebPageTest & Real-User Metrics

Measure before and after changes. Use Lighthouse in Chrome DevTools for lab metrics (LCP, FID, CLS). Use WebPageTest for network waterfall and advanced analysis. Collect Real User Monitoring (RUM) to see true user experience.

Key metrics to know

  • LCP — Largest Contentful Paint (speed of main content render)
  • FID — First Input Delay (interactivity)
  • CLS — Cumulative Layout Shift (visual stability)
  • TTFB — Time To First Byte (server responsiveness)

Tip: Optimize for LCP by ensuring hero image and main heading load early (preload image, critical CSS).

15 — Accessibility helps SEO

Accessible pages are usually better indexed. Use semantic HTML, correct alt attributes, ARIA where needed, and ensure content is reachable without JavaScript. 

<img src="/images/team.jpg" alt="Design team planning user interfaces">
<nav aria-label="primary navigation">...</nav>

16 — Pre-Launch Checklist

  1. Unique page titles and meta descriptions for all pages.
  2. Sitemap.xml present and submitted to Google Search Console.
  3. robots.txt configured correctly.
  4. All images optimized and using responsive srcset or <picture>.
  5. Fonts loaded with preconnect or preloads; minimal font variants used.
  6. CSS critical path optimized and main stylesheet non-blocking.
  7. JS deferred; third-party scripts loaded async.
  8. Compression (gzip/brotli) enabled on server and caching headers set.
  9. HTTPS enforced and HTTP/2 or HTTP/3 enabled if available.
  10. Lighthouse score checked and common issues resolved.

17 — Common Problems & Quick Fixes

Slow LCP

Preload hero images, inline critical CSS, reduce server response time (optimize backend), and use CDN.

Large JavaScript bundle

Split code, remove unused libraries, and lazy-load non-critical components.

Fonts cause flash of invisible text (FOIT)

Use display=swap or preload critical fonts, or self-host smaller subsets.

Pages not being crawled

Check robots.txt, ensure pages aren’t blocked by meta robots noindex, and submit sitemap to Search Console.

Conclusion

Optimizing for SEO and speed is an iterative process. Make measurable changes, test frequently, and keep the user experience central. A fast, accessible site not only ranks better but converts visitors into users and customers more effectively.

If you follow the practices in this lesson — meaningful metadata, structured data, responsive images, smart font loading, caching, compression, and measurement — you’ll have a solid foundation to build professional, high-performing websites.
